## Contexte

Les 8 types `ParticleSystemFactory` (0x15800), `ParticleSystem` (0x15801),
`BaseEmitter` (0x15805), `SpriteEmitter` (0x15806), `ParticleAnimation` (0x15808),
`EmitterAnimation` (0x15809), `GeneratorAnimation` (0x1580A) et
`ParticleInstancingInfo` (0x1580B) sont présents dans l'enum `ChunkType` et dans
`operator<<`, mais **absents de `p3d.json`** : aucune classe générée, aucun parsing.
Indispensable pour fumée / feu / nitro (chunk monde `InstParticleSystem = 0x3001001`).

## Approche

Workflow codegen : éditer `dev/codegen/p3d.json` → lancer `dev/codegen/run.bat`
(`donut-codegen.exe`) → les classes apparaissent dans `P3D.generated.{h,cpp}`.

1. Reverse les 8 layouts (Ghidra sur `radcore` / pure3d, ou recouper avec des dumps de
   `.p3d` connus contenant des particules). Décrire chaque chunk dans `p3d.json` avec la
   grammaire en place : `u32`, `float`, `vec3`, `string`, `string[4]` (FourCC),
   `child X`, `children X`, `buffer<T[n]>`.
2. Hiérarchie attendue : `ParticleSystemFactory` contient `ParticleSystem` ; un
   `ParticleSystem` agrège des émetteurs (`BaseEmitter` → `SpriteEmitter`) et leurs
   animations (`ParticleAnimation`, `EmitterAnimation`, `GeneratorAnimation`) +
   `ParticleInstancingInfo`.
3. Régénérer, puis vérifier les classes/`Load()` (modèle `Camera` / `MultiController`
   dans `P3D.generated.cpp`).
4. Brancher au minimum la lecture dans `Level.cpp` (cas `InstParticleSystem`) sans encore
   exiger le rendu GPU.

## Critères d'acceptation

- [ ] Les 8 chunks sont décrits dans `p3d.json` avec leurs champs reversés.
- [ ] Après `run.bat`, `P3D.generated.h` expose `class ParticleSystem` + 7 autres avec getters.
- [ ] Un `.p3d` contenant des particules se parse sans assert ni dépassement de `MemoryStream`.
- [ ] Le dispatch `Level.cpp` reconnaît `InstParticleSystem` (au moins logué).
