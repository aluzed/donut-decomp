# P3D-017 — Images non-PNG (palettisées, DXT) dans ImageDecoder::Decode

- **Status:** TODO
- **Priority:** P3
- **Module:** P3D
- **Depends on:** —
- **Files:** `src/P3D/P3DChunk.cpp:6-21` (`ImageDecoder::Decode`), `src/P3D/P3DChunk.h:262-270` (struct `ImageDecoder`), `dev/codegen/p3d.json:427-442` (chunk `Image`), `src/Render/Texture.cpp`

## Contexte

`ImageDecoder::Decode` (`P3DChunk.cpp`) ne gère **que le PNG** : `stb_image` est compilé
avec `#define STBI_ONLY_PNG` et le commentaire le dit explicitement (« just does png for
now »). Les autres formats stockés dans les chunks `Image` (0x19001) — **palettisé** et
**DXT/S3TC** — ne sont pas décodés. Le chunk `Image` expose pourtant déjà `palettized u32`
et `format u32` (cf. `p3d.json`), donc le format est connu à la lecture.

## Approche

1. Recenser les `format` réellement présents (lire `Image::GetFormat()` /
   `Image::GetPalettized()` sur des `.p3d` de textures).
2. Dispatcher dans `ImageDecoder::Decode` selon le format :
   - **PNG** : chemin actuel `stbi_load_from_memory` (conserver) ;
   - **Palettisé** : décoder palette + indices → RGBA (remplir `width`/`height`/`comp`/`data`) ;
   - **DXT1/DXT3/DXT5** : décompresser les blocs S3TC → RGBA (implémentation maison ou
     lib légère ; pas de dépendance lourde).
3. La signature `ImageDecoder Decode(const std::vector<uint8_t>&)` ne reçoit pas le format ;
   ajouter un paramètre format/palette (ou surcharge) en s'appuyant sur les champs du chunk
   `Image`. Adapter les appelants (`src/Render/Texture.cpp`).
4. Conserver le fallback PNG pour ne rien casser.

## Critères d'acceptation

- [ ] `ImageDecoder::Decode` décode au moins palettisé + DXT en plus du PNG.
- [ ] Le format est déterminé depuis le chunk `Image` (`format`/`palettized`), pas deviné.
- [ ] Une texture DXT et une texture palettisée réelles s'affichent correctement.
- [ ] Les textures PNG existantes continuent de fonctionner (aucune régression).
