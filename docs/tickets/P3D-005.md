- **Status:** TODO

## Contexte

Chunks de texte localisé du frontend, état inégal :

- `FrontendStringHardCoded` (0x1800C) : `name string`, `text string` — **déjà dans
  `p3d.json`** mais non régénéré ;
- `FrontendTextBible` (0x1800D) : `name string`, `language string`, `numStrings u32` —
  **déjà dans `p3d.json`** mais **incomplet** (les `numStrings` entrées clé→texte ne sont
  pas décrites) et non régénéré ;
- `FrontendLanguage` (0x1800E), `FrontendObjectResource` (0x18101),
  `FrontendTextStyleResource` (0x18104), `FrontendTextBibleResource` (0x18105) :
  déclarés dans l'enum `ChunkType` mais **absents de `p3d.json`** → à reverser.

`FrontendStringTextBible` (0x1800B, `name`/`key`) est déjà géré et sert de référence pour
la résolution clé→texte.

## Approche

1. Compléter `FrontendTextBible` dans `p3d.json` : ajouter les `numStrings` paires
   (probablement `children FrontendStringHardCoded` ou un `buffer` de paires
   `key string`/`value string`) — reverser le layout exact.
2. Décrire `FrontendLanguage`, `FrontendObjectResource`, `FrontendTextStyleResource`,
   `FrontendTextBibleResource` (reverse). Les `*Resource` suivent le motif des resources
   frontend existants (`FrontendImageResource` : `name`/`version`/`filepath`, ligne 581).
3. Lancer `dev/codegen/run.bat`, vérifier les `Load()`.
4. Brancher dans `src/FrontendProject.cpp` pour résoudre les clés de texte vers la chaîne
   localisée selon la langue courante (base nécessaire à **UI-LANG**).

## Critères d'acceptation

- [ ] `FrontendTextBible` complété (entrées clé→texte parsées).
- [ ] `FrontendStringHardCoded`, `FrontendLanguage`, `FrontendObjectResource`, `FrontendTextStyleResource`, `FrontendTextBibleResource` décrits et générés.
- [ ] Une clé de texte se résout vers la chaîne correcte pour une langue donnée.
- [ ] Le frontend se parse sans assert sur un `.p3d` localisé réel.
